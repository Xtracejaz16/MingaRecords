# ADR 004: Revisión de Stack Tecnológico y Arquitectura del Backend

- **Estado:** Aceptado.
- **Fecha:** 15 de mayo de 2026.
- **Autor:** Sebastián Estrada y Yair Santiago Cetre.

## 1. Contexto

El SDD original del backend (v1.0, 12 de mayo 2026) propuso una stack basada en Fastify, Stripe, Railway y BunnyCDN con un costo estimado de ~$18/mes y 4 microservicios separados. Durante la revisión del plan se identificaron restricciones reales que no estaban reflejadas:

1. **Presupuesto $0 absoluto** — No hay dinero para infraestructura en fase MVP.
2. **Experiencia del equipo en Express** — Ambos developers dominan Express.js; Fastify requeriría aprendizaje.
3. **Stripe no disponible en LATAM** — No podemos operar legalmente con Stripe desde nuestro país sin empresa en EE.UU./Europa.
4. **Deadline de 1 semana** — No hay margen para curvas de aprendizaje ni setups complejos.
5. **ADR 001 revisado** — Microservicios como estilo arquitectónico se mantiene, pero la implementación física para MVP es monolito modular.

## 2. Decisiones

### 2.1 Framework: Express.js sobre Fastify

Express.js será el framework backend para el monolito.

**Razones:**
- El equipo ya domina Express; Fastify requeriría 2-4 horas de aprendizaje que no tenemos con deadline de 1 semana.
- La diferencia de performance (20k vs 60k req/s) es irrelevante para un MVP con < 1000 usuarios.
- La arquitectura hexagonal funciona igual con ambos — el framework es solo un adaptador HTTP.
- Zod como validador da mejores mensajes de error en español que JSON Schema nativo de Fastify.

### 2.2 Pasarela de Pagos: MercadoPago sobre Stripe

MercadoPago será la pasarela de pagos principal para el MVP.

**Razones:**
- Stripe no está disponible en nuestro país sin empresa registrada en país soportado.
- MercadoPago tiene sandbox completo, SDK Node.js oficial (`mercadopago` npm), y está disponible en Colombia y Argentina.
- Ofrece las mismas capacidades: checkout con redirect, webhooks, reembolsos.
- Sin costo fijo mensual, solo ~5% de comisión por venta.

**Arquitectura:** El puerto `PaymentGateway` en el domain permanece sin cambios. Solo se crea un nuevo adaptador `MercadoPagoGateway` en `infrastructure/adapters/`.

### 2.3 Simplificaciones de Infraestructura para $0

Se eliminan los siguientes componentes del MVP:

| Componente | Razón de eliminación |
|------------|---------------------|
| **BunnyCDN** | R2 ya usa la red de Cloudflare; trial de 14 días luego paga |
| **Upstash Redis** | Con 1 proceso, rate limiting en memoria es suficiente |
| **Cloudflare Workers** (API Gateway) | CORS directo en Express es suficiente |
| **Circuit breaker** | Se reintroduce cuando haya múltiples instancias |

### 2.4 Arquitectura: Monolito Modular sobre Microservicios

Para el MVP, el backend se implementa como **1 monolito modular** con estructura hexagonal por módulo, en vez de 4 microservicios separados.

```
apps/backend/
├── src/modules/
│   ├── auth/         → domain/, application/, infrastructure/
│   ├── catalog/      → domain/, application/, infrastructure/
│   ├── streaming/    → domain/, application/, infrastructure/
│   └── payments/     → domain/, application/, infrastructure/
├── src/shared/       → middleware, tipos, utils
└── src/app.ts        → Express app
```

**Razones:**
- 2 developers con deadline de 1 semana: no hay margen para gestionar 4 deploys independientes.
- $0 presupuesto: no hay infraestructura gratuita que soporte 4 servicios 24/7.
- La estructura hexagonal por módulo hace la extracción a microservicios trivial en v2.
- Sin overhead de comunicación HTTP entre servicios: llamadas directas a casos de uso.
- Transacciones ACID abarcan múltiples módulos (ej: pago + actualizar ventas en 1 transacción).

**Lo que NO cambia:**
- Cada módulo mantiene su `domain/`, `application/`, `infrastructure/` separados.
- Los puertos (interfaces) definen los límites entre módulos.
- La extracción a microservicios en v2 es: copiar módulo → agregar HTTP → deploy separado.

### 2.5 Infraestructura de Deploy: AWS Free Tier (12 meses)

AWS free tier será la plataforma de deploy para el MVP:

- **EC2 t3.micro**: 750h/mes gratis (1 instancia 24/7) por 12 meses. Corre el monolito con Docker.
- **Lambda + API Gateway**: 1M requests/mes gratis permanente. Alternativa serverless.

La decisión entre EC2 y Lambda se evalúa en la marcha.

## 3. Alternativas consideradas

| Alternativa | Por qué se descartó |
|-------------|---------------------|
| **Fastify** | Curva de aprendizaje innecesaria con deadline de 1 semana |
| **Stripe** | No disponible legalmente en nuestro país |
| **4 microservicios separados** | Costo ~$18/mes, setup complejo, overhead para 2 devs |
| **Oracle Cloud** | Setup demasiado complejo (1-2 semanas) para el deadline |
| **Railway** | $5 crédito inicial se agota en 1 mes; no es free permanente |
| **Hono** | Framework nuevo para el equipo; Express ya lo sabemos |

## 4. Consecuencias

**Positivas:**
- Time-to-market máximo: Express ya lo sabemos, MercadoPago sandbox está listo.
- Costo $0 en MVP (excepto dominio $10/año).
- La arquitectura hexagonal protege contra futuros cambios de pasarela o framework.
- Menos componentes = menos superficie de bugs.
- Transacciones ACID entre módulos (imposible con microservicios separados).

**Negativas:**
- AWS free tier expira en 12 meses (plan de migración a Oracle Cloud Always Free como backup).
- MercadoPago tiene menor adopción global que Stripe (no afecta LATAM).
- Sin Redis en MVP: rate limiting menos preciso, sin cache compartido.
- Sin CDN dedicado: R2 directo tiene latencia ligeramente mayor que BunnyCDN.
- Un solo punto de falla: si el proceso Express cae, caen todos los módulos.
